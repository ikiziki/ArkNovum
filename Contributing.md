# Contributing to ArkNovum

Thank you for your interest in contributing to ArkNovum.

ArkNovum exists because hobbies are built by communities. Whether you are writing code, designing interfaces, documenting features, creating plugins, improving data sources, or sharing knowledge about a hobby, your contribution helps build a better platform for everyone.

---

# Ways To Contribute

There are many ways to contribute to ArkNovum.

## Development

Developers can contribute by:

- Building core platform features
- Improving performance
- Fixing bugs
- Creating plugins
- Improving APIs
- Adding integrations
- Improving security
- Writing automated tests

---

## Plugin Development

Plugins are one of the most important parts of ArkNovum.

Community members are encouraged to create plugins for hobbies including:

- Model building
- RC vehicles
- Miniatures
- Model railroads
- Collecting
- Crafting
- Electronics
- Photography
- Other hobby communities

Plugins should follow the ArkNovum plugin architecture and should remain focused on extending the platform without modifying the core system unnecessarily.

---

## Documentation

Good documentation is essential.

Contributors can help with:

- Installation guides
- User guides
- Plugin documentation
- API documentation
- Tutorials
- Examples
- Troubleshooting guides

---

## Design

ArkNovum is designed for hobbyists first.

Design contributions are welcome for:

- User interface improvements
- Accessibility
- Themes
- Icons
- Branding
- User experience improvements

---

## Testing

Testing contributions help ensure ArkNovum remains reliable.

You can help by:

- Testing new releases
- Reporting bugs
- Testing plugins
- Validating documentation
- Reporting usability issues

---

# Before Contributing

Before starting significant work:

1. Review existing issues and discussions.
2. Check whether someone is already working on the same feature.
3. Open a discussion for major changes.
4. Make sure the proposed change aligns with ArkNovum's goals.

Large changes should be discussed before development begins.

---

# Development Philosophy

ArkNovum follows several guiding principles.

## Preserve Modularity

The core platform should remain hobby-agnostic.

If a feature only applies to one hobby, it likely belongs in a plugin.

---

## Favor Extensibility

When adding functionality, consider:

- Can another plugin use this?
- Can this be exposed through the API?
- Does this belong in the core?

---

## Respect User Data

User collections, projects, photos, and history are valuable.

Contributions should prioritize:

- Data integrity
- Backward compatibility
- Safe migrations
- Exportability

---

## Keep Hobbyists In Mind

ArkNovum is not designed only for technical users.

Features should be approachable for people who simply want to document what they love.

---

# Plugin Contribution Guidelines

Plugins should:

- Have clear documentation
- Define their purpose
- Avoid unnecessary dependencies
- Follow ArkNovum coding standards
- Respect user privacy
- Avoid modifying unrelated core functionality

A good plugin should be:

- Easy to install
- Easy to disable
- Easy to remove
- Safe to update

---

# Code Standards

All contributions should:

- Use clear naming conventions
- Include documentation where appropriate
- Include tests when possible
- Avoid unnecessary complexity
- Follow existing project structure

Readable and maintainable code is preferred over clever code.

---

# Reporting Bugs

When reporting a bug, include:

- ArkNovum version
- Operating system
- Installation method
- Plugin versions involved
- Steps to reproduce
- Expected behavior
- Actual behavior
- Relevant logs or screenshots

A good bug report helps resolve issues faster.

---

# Feature Requests

Feature requests are welcome.

Before submitting a feature request, consider:

- Does this benefit many users?
- Does this belong in the core or a plugin?
- Does this maintain ArkNovum's modular philosophy?

Include:

- The problem being solved
- The proposed solution
- Possible alternatives
- Who would benefit from the feature

---

# Pull Requests

Pull requests should:

- Have a clear purpose
- Reference related issues when applicable
- Include documentation updates if needed
- Include testing information
- Avoid unrelated changes

Pull request descriptions should explain:

- What changed
- Why it changed
- How it was tested

---

# Community Plugins

Community-created plugins are encouraged.

Community plugins should clearly state:

- Plugin purpose
- Supported versions
- Installation instructions
- Maintainer information
- License information

Official ArkNovum branding should not be used to imply endorsement unless the plugin is officially supported.

---

# Commercial Plugins

ArkNovum may offer official Pro plugins in the future.

Community developers may create their own plugins, but commercial plugins must:

- Clearly identify ownership
- Respect ArkNovum branding guidelines
- Follow the project contribution and licensing requirements

---

# Code of Conduct

ArkNovum is built around shared interests and community collaboration.

Contributors are expected to:

- Be respectful
- Welcome different skill levels
- Provide constructive feedback
- Help others learn
- Keep discussions focused on improving the project

Everyone should feel welcome regardless of technical experience.

---

# Getting Started

New contributors can begin by:

1. Reading the project documentation
2. Reviewing open issues
3. Joining discussions
4. Creating a small improvement
5. Building a plugin
6. Helping improve documentation

Every contribution matters.

---
